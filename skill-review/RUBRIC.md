# Skill Review Rubric — 详细评分锚定

## D1: 触发精度 (Trigger Precision) — 权重 1.5x

**核心问题**: description 够不够准确？会不会被误触发或漏触发？

| 分 | 锚定描述 |
|----|---------|
| 1 | description 缺失或极度模糊（如 "Helps with documents"），agent 无法判断何时触发 |
| 2 | description 只有功能描述，没有触发条件（"Use when..." 不明确） |
| 3 | description 有触发条件但只有 1-2 个关键词，容易漏触发 |
| 4 | description 覆盖主要触发场景，包含具体中英文触发词，边界清晰 |
| 5 | description 能区分"应该触发"和"不该触发"的场景，列举了反例或排除条件 |

**检查清单**:
- [ ] `description` 包含 "Use when [specific triggers]" 句式？
- [ ] 有中文触发词吗？（如果目标用户用中文）
- [ ] 会不会被类似但不相关的请求误触发？
- [ ] 如果用户换一种说法，还能匹配到吗？

---

## D2: 渐进披露 (Progressive Disclosure) — 权重 1.0x

**核心问题**: 主文件是否简洁，细节是否合理外置？

| 分 | 锚定描述 |
|----|---------|
| 1 | SKILL.md > 500 行，无拆分，所有内容挤在一起 |
| 2 | SKILL.md 200-500 行，无拆分但有章节 |
| 3 | SKILL.md < 200 行，有引用外部文件但不系统 |
| 4 | SKILL.md < 100 行，模板/参考/示例独立文件，引用清晰 |
| 5 | < 100 行且每个外部文件被引用的场景明确（"只在需要 X 时读 Y"） |

**检查清单**:
- [ ] SKILL.md 是否 < 100 行？
- [ ] 模板/参考/示例是否在独立文件中？
- [ ] 外部文件是否被明确告知"什么时候才需要读"？

---

## D3: 负空间 (Negative Space) — 权重 1.0x

**核心问题**: 有没有定义"不做什么"？有没有边界？

| 分 | 锚定描述 |
|----|---------|
| 1 | 只有正面指令，没有边界定义 |
| 2 | 有模糊的"注意"提示但无具体排除条件 |
| 3 | 定义了什么时候不适合使用（如"单行拼写错误不必记录"） |
| 4 | 既有边界条件又有量化标准（如"修复涉及 ≥2 层联动才记录"） |
| 5 | 负空间定义完整且可操作——agent 能自主判断"这次不该用" |

**检查清单**:
- [ ] 有没有定义"什么时候不要用这个 skill"？
- [ ] 有没有量化判断标准（不是"sometimes"而是"≥2 files"）？
- [ ] agent 能不能在没有用户帮助下做出"不该触发"的判断？

---

## D4: 输出契约 (Output Contract) — 权重 1.0x

**核心问题**: 每步有没有可验证的产物？用户能不能判断"完成了"？

| 分 | 锚定描述 |
|----|---------|
| 1 | 没有明确的输出格式或产物定义 |
| 2 | 有模糊的输出描述（"生成一个报告"）但无格式要求 |
| 3 | 定义了输出格式和位置，但无验证步骤 |
| 4 | 每步有可检查的产物，用户能判断"是否完成" |
| 5 | 包含自验证步骤（如"写入后检查文件是否存在"），输出可被下游消费 |

**检查清单**:
- [ ] 每一步的输出是什么？能不能用一条命令验证？
- [ ] 输出格式是否明确（模板、JSON schema 等）？
- [ ] 有没有 ⛔ BLOCKING 节点（必须用户确认才能继续）？

---

## D5: 环境独立 (Environment Independence) — 权重 0.5x

**核心问题**: 有没有硬编码路径？能否在别人的机器上工作？

| 分 | 锚定描述 |
|----|---------|
| 1 | 硬编码了 ≥3 个绝对路径，换台机器无法运行 |
| 2 | 硬编码了 1-2 个路径或用户名 |
| 3 | 使用相对路径和 `~/.claude/`，但假设了特定的 vault 结构 |
| 4 | 路径通过参数或询问用户获取，有合理的默认值 |
| 5 | 零硬编码，只用标准路径（`~/.claude/`, `./`, 用户指定），可跨平台 |

**检查清单**:
- [ ] 有没有 `C:\Users\<name>` 或 `/home/<name>` ？
- [ ] 默认目录是否通用（不是 `10_Projects/AI工程笔记/`）？
- [ ] 如果换一个人用，需要改几处才能跑？

---

## D6: 失败模式 (Failure Modes) — 权重 1.0x

**核心问题**: 出错了会怎样？有没有覆盖边界情况？

| 分 | 锚定描述 |
|----|---------|
| 1 | 没有错误处理，异常路径完全未定义 |
| 2 | 提到了"如果出错"但没有具体策略 |
| 3 | 定义了 1-2 种常见失败场景的处理方式 |
| 4 | 覆盖了主要失败场景（同名文件、空输入、权限不足等）+ 降级方案 |
| 5 | 有完整的错误分类（重试 / 降级 / 升级 / 阻塞），每种有明确触发条件和退出路径 |

**检查清单**:
- [ ] 如果目标文件已存在，会覆盖还是报错？
- [ ] 如果输入为空，会怎样？
- [ ] 如果中间步骤失败，会回滚还是继续？
- [ ] 有没有"连续失败 N 次就停止"的上限？

---

## D7: 组合性 (Composability) — 权重 0.5x

**核心问题**: 能不能被其他 skill 调用？有没有清晰的 I/O 边界？

| 分 | 锚定描述 |
|----|---------|
| 1 | 完全独立，不与其他 skill 产生任何关联 |
| 2 | 理论上可以被调用但未定义 I/O 接口 |
| 3 | 有明确的输入参数和输出格式，但未被其他 skill 实际引用 |
| 4 | 定义了与其他 skill 的连接点（如"diagnose 完成后建议调用 postmortem-note"） |
| 5 | 多个 skill 形成调用链，每个 I/O 明确，可替换实现 |

**检查清单**:
- [ ] 如果其他 skill 想调用它，需要传什么？返回什么？
- [ ] 有没有 `related` 字段指向相关 skill？
- [ ] 有没有定义"在某个 skill 完成后自动建议"的连接点？

---

## 评分快捷参考

```
D1: 描述是否精确 → 读 SKILL.md 第 2-4 行
D2: 行数 + 文件数 → wc -l SKILL.md + ls
D3: 有没有 "不做什么" → grep "不要|不必|不适合|exclud"
D4: 每步有没有产物 → 找 "输出|保存|创建|生成|Write"
D5: 有没有硬编码 → grep "C:\\|/home/|/Users/"
D6: 有没有错误处理 → grep "失败|错误|覆盖|冲突|重试"
D7: 有没有被引用 → grep "postmortem|diagnose|article|调用"
```
